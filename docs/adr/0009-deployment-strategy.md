# ADR-0009: Monorepo GitOps Deployment Strategy via ArgoCD Image Updater

Date: 2026-06-17

## Status

Accepted

## Context

Following the architecture established in `ADR-0001`, our RAG infrastructure consists of decoupled service components (`apps/api`, `apps/chunker`, `apps/indexer`) targeted for deployment inside an AWS EKS cluster. In conventional production environments, application source code and infrastructure manifests are decoupled into separate Git repositories to enforce strict RBAC boundaries.

However, for this open-source architecture, enforcing a multi-repository layout introduces significant operational liabilities: it degrades the Developer Experience (DX), inflates cognitive load for external contributors, and breaks the atomic visibility of the system for external architectural review.

To maintain a unified Monorepo structure while achieving fully automated Continuous Deployment (CD), we require a mechanism that synchronizes container registry states with the Kubernetes cluster configuration without:
1. Orchestrating brittle GitHub Actions workflows that require administrative Git write-access and introduce commit-loop vulnerabilities.
2. Contaminating core declarative manifests (`values.yaml`, `Deployment` charts) with dynamic, short-lived commit-hash tags.
3. Incurring unnecessary infrastructure expenditures (Total Cost of Ownership - TCO) through cloud provider container registry storage and egress fees during the validation phase.

## Decision

We adopt an in-cluster GitOps deployment paradigm utilizing **ArgoCD Image Updater** combined with **GitHub Container Registry (GHCR)**, governed by the following architectural mandates:

1. **Public Container Registry Infrastructure (GHCR):** We explicitly bypass cloud-provider-managed registries (AWS ECR) for public distribution. OCI images for all three internal services are compiled via GitHub Actions and published to `ghcr.io`. This eliminates infrastructure egress overhead and facilitates public image availability without complex Kubernetes `imagePullSecrets` lifecycle management.
2. **Strict CI/CD Separation of Concerns:** The GitHub Actions CI pipeline is confined strictly to asset compilation, static analysis, test execution, and OCI image publication tagged with the short Git commit SHA (`sha-${{ github.sha }}`). The CI layer possesses zero knowledge of destination environment topology and holds no write mutations over the Git repository.
3. **In-Cluster State Synchronization:** Deployment automation is delegated entirely to the ArgoCD Image Updater controller operating natively within the AWS EKS cluster. The controller continuously polls the GHCR API to resolve image tags against defined synchronization policies.
4. **Declarative Git Write-Back (`write-back-method: git`):** To preserve the foundational GitOps principle of "Git as the Single Source of Truth," the controller is configured to execute state persistence via the `git` write-back method.
    * Upon identifying a state delta in GHCR, the in-cluster controller leverages a tightly scoped, localized Git credential to execute an automated commit directly back to the `master` branch.
    * **State Isolation:** The controller is strictly prohibited from mutating primary Helm values or base Kubernetes templates. Instead, it generates or overrides an isolated tracking artifact named `.argocd-source-rag.yaml` within the concrete environment scope (`deploy/k8s/environments/local`).

## Consequences

* **Unified Repository Cohesion:** External engineers and reviewers can clone a singular repository to audit the entire operational footprint—spanning application logic, CI workflow automation, and declarative GitOps deployment state.
* **Audit Trail Traceability:** Automated commits generated by the in-cluster controller (`argocd-bot`) are isolated exclusively to the `.argocd-source-*.yaml` files. These automated diffs can be instantly filtered out via standard Git CLI flags (`--invert-grep`) or Git UI configurations, keeping the human engineering history pristine.
* **Disaster Recovery and State Drift Protection:** In the event of catastrophic cluster degradation, Git remains the absolute source of truth. Re-applying the root ArgoCD Application forces the engine to parse the `.argocd-source-*.yaml` manifests, instantly restoring the cluster state to the exact image tags running prior to the failure, mitigating the "blind update" risks inherent to in-memory CD configurations.
* **Deterministic Remediation (Rollbacks):** Under critical operational failures, engineers can execute a native `git revert` on the specific automation commit or implement a temporary manual synchronization block via the ArgoCD control plane, placing the application in a controlled `OutOfSync` state to lock configuration to a safe historical milestone.
* **API Rate-Limiting Controls:** Polling intervals against the GHCR API must be tuned via global controller configurations to prevent upstream rate-limiting, particularly during intensive parallel development cycles across the monoreпо components.
